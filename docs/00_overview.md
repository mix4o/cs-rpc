# 全体設計書 — cs-rpc

クロスプラットフォーム（Windows / Linux）で動作する、クライアント・サーバ間の
汎用リクエスト応答（RPC）プログラムの全体設計。

- 関連: [クライアント設計書](01_client_go.md) / [サーバ設計書](02_server_python.md) / [プロトコル&コマンド仕様](03_protocol_and_commands.md)

> 本書は基盤となる同期 RPC（`/rpc`）の設計。その後に追加した**コントロールプレーン**
> （ジョブキュー・ワーカ実行・進捗/キャンセル・コマンドカタログ）は
> [03_protocol_and_commands.md](03_protocol_and_commands.md) にまとめている。

---

## 1. 目的とスコープ

### 1.1 目的
クライアントから「コマンド（メソッド）＋パラメータ」をサーバへ送信し、
サーバ側で処理を実行して結果を返す、**汎用RPC基盤**を構築する。

特定業務に依存しない土台とし、サーバ側にコマンドを追加していくだけで
機能拡張できることを重視する（サーバは Python で「変更しやすさ」を優先）。

### 1.2 スコープ
| 含む | 含まない（初期版） |
| --- | --- |
| HTTP/JSON によるリクエスト応答 | 認証・認可（※後付けできる設計にする） |
| 単発の同期リクエスト | 双方向ストリーミング・サーバpush |
| コマンドのディスパッチと結果返却 | 大容量ファイル転送の最適化 |
| エラー体系・タイムアウト・リトライ | 分散/クラスタリング・負荷分散 |
| クロスプラットフォームなクライアント配布 | GUI |

### 1.3 想定環境
- クライアント: Windows / Linux（単一バイナリで配布したい） → **Go**
- サーバ: Linux 中心（Windows でも起動可能） → **Python**
- ネットワーク: まずはローカル / 信頼された内部ネットワーク（認証なし）

## 2. アーキテクチャ概要

```
┌────────────────────┐         HTTP POST /rpc          ┌────────────────────┐
│   Client (Go)      │  ─────  JSON Request  ───────▶  │   Server (Python)  │
│                    │                                 │                    │
│  CLI / ライブラリ  │  ◀────  JSON Response  ────────  │  Dispatcher +      │
│  RPC Client        │                                 │  Command Handlers  │
└────────────────────┘                                 └────────────────────┘
```

- 通信方式: **HTTP/1.1 + JSON**（REST的な単一エンドポイント）
- プロトコル: **JSON-RPC 2.0 準拠**（後述）。広く知られた仕様で、
  ライブラリ・デバッグツールが豊富。`curl` でも叩ける。
- 同期モデル: 1リクエスト = 1レスポンス（HTTP のリクエスト/レスポンスに対応）

### 2.1 なぜこの構成か
- **HTTP/JSON**: Windows/Linux 双方でツール・ライブラリが成熟。プロキシ・ログ・
  デバッグが容易で、後から TLS / 認証ヘッダを足しやすい。
- **JSON-RPC 2.0**: 「method + params → result / error」という RPC の意味論に素直。
  独自プロトコルを発明せずに済み、エラーコード体系も規定済み。
- **Go クライアント**: 依存なしの単一バイナリを Windows/Linux 向けにクロスコンパイル可能。
- **Python サーバ**: ハンドラ追加が容易。FastAPI で自動ドキュメント・型検証を活用。

## 3. 通信プロトコル仕様

### 3.1 エンドポイント
| メソッド | パス | 用途 |
| --- | --- | --- |
| `POST` | `/rpc` | RPC 本体。JSON-RPC リクエストを受ける |
| `GET`  | `/healthz` | ヘルスチェック（死活監視・疎通確認） |
| `GET`  | `/rpc/methods` | 登録済みメソッド一覧（開発時の自己記述用） |

上記はデータプレーン（同期 RPC）。この他に**コントロールプレーン**（`GET /`
コントロールページ、`POST /control/*` のジョブキュー/ワーカ用 API）がある。
エンドポイント一覧とコマンド/パラメータの詳細は
[03_protocol_and_commands.md](03_protocol_and_commands.md) を参照。

- Content-Type: `application/json; charset=utf-8`
- 文字コードは **UTF-8 固定**（Windows/Linux 間の文字化け防止）

### 3.2 リクエスト形式（JSON-RPC 2.0）
```json
{
  "jsonrpc": "2.0",
  "id": "b6c1e2f0-...",         // 相関ID（クライアント生成のUUID）。通知の場合は省略
  "method": "echo",             // 実行するコマンド名
  "params": { "message": "hi" } // パラメータ（オブジェクト or 配列）
}
```

### 3.3 レスポンス形式
成功:
```json
{
  "jsonrpc": "2.0",
  "id": "b6c1e2f0-...",
  "result": { "message": "hi" }
}
```

エラー:
```json
{
  "jsonrpc": "2.0",
  "id": "b6c1e2f0-...",
  "error": {
    "code": -32601,
    "message": "Method not found",
    "data": { "method": "unknown_cmd" }
  }
}
```

### 3.4 エラーコード体系
JSON-RPC 2.0 標準コードを踏襲し、アプリ固有エラーは予約域外を使う。

| code | 意味 | 発生例 |
| --- | --- | --- |
| `-32700` | Parse error | JSON として壊れている |
| `-32600` | Invalid Request | 必須フィールド欠落・型不正 |
| `-32601` | Method not found | 未登録メソッド |
| `-32602` | Invalid params | パラメータの型/必須不足 |
| `-32603` | Internal error | ハンドラ内で未捕捉の例外 |
| `-32000`〜`-32099` | Server error（予約） | 認証失敗等（将来） |
| `1000`〜 | アプリ固有エラー | 各コマンド定義のドメインエラー |

### 3.5 HTTP ステータスとの対応
- 正常に JSON-RPC 応答（result/error どちらでも）を返せる → **200 OK**
- ボディが JSON として壊れている → 400（body は Parse error の JSON-RPC エラー）
- サーバ内部で応答生成すら不能 → 500
- 認証（将来）失敗 → 401 / 403

> 方針: **アプリレベルの失敗は 200 + error オブジェクト**、
> **トランスポート/プロトコルレベルの失敗は 4xx/5xx** に寄せる。

### 3.6 バージョニング
- プロトコルの破壊的変更に備え、パスに版を持たせる余地を残す（例: `/v1/rpc`）。
  初期版は `/rpc`（= v1 相当）とし、v2 が必要になった時点で並存させる。
- メソッド単位の互換は `method` 名（例: `job.run`）と `params` の追加で吸収する。

## 4. 代表シーケンス

### 4.1 正常系
```
Client                         Server
  |   POST /rpc {method,params}   |
  |------------------------------▶|
  |                               |  parse → validate → dispatch("echo")
  |                               |  handler 実行
  |   200 {result}                |
  |◀------------------------------|
```

### 4.2 異常系（未登録メソッド）
```
Client                         Server
  |   POST /rpc {method:"nope"}   |
  |------------------------------▶|
  |                               |  dispatch → 見つからない
  |   200 {error:-32601}          |
  |◀------------------------------|
```

### 4.3 タイムアウト
- クライアントは接続・応答にタイムアウトを設定（既定 30s、コマンド毎に上書き可）。
- タイムアウト時はクライアント側でリトライ判断（冪等なメソッドのみ再送）。

## 5. 横断的関心事

| 項目 | 方針 |
| --- | --- |
| 文字コード | UTF-8 固定。改行は本文中では LF（HTTP ヘッダは CRLF は HTTP 実装任せ） |
| 相関ID | クライアントが UUID を採番し `id` に設定。ログ突合に使用 |
| ロギング | 両者で `id` を出力。サーバは method/所要時間/結果コードを構造化ログ |
| タイムアウト | クライアント既定 30s。サーバもハンドラ実行に上限を設ける |
| リトライ | クライアント側。冪等メソッドのみ指数バックオフで最大N回 |
| サイズ制限 | リクエストボディ上限（既定 10MB）。超過は 413 相当のエラー |
| 設定 | 環境変数 > 設定ファイル > 既定値 の優先順（両者共通の考え方） |

## 6. セキュリティ（将来拡張の余地）

初期版は認証なし（ローカル/信頼環境）。ただし後付けを容易にするため:

- **トランスポート**: リバースプロキシ（nginx/caddy）で TLS 終端、
  もしくはサーバに TLS を直付けできる構成にしておく。
- **認証フック**: サーバのミドルウェア層に「認証チェックの差し込み口」を用意
  （初期は素通し）。`Authorization` ヘッダや APIキーを後から検証できる。
- クライアントは任意ヘッダを付与できる設計にしておく（トークン注入用）。

詳細は各設計書の「拡張ポイント」を参照。

## 8. 品質・テスト方針
- **契約テスト**: プロトコル（3章）の JSON 例をフィクスチャ化し、両者で検証。
- サーバ: ハンドラ単体テスト + ディスパッチャの結合テスト。
- クライアント: モックサーバに対する結合テスト（正常/エラー/タイムアウト）。
- クロスプラットフォーム: Go は Windows/Linux 双方でビルド・実行を CI で確認。
